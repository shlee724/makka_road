## 0. 이 문서의 성격

맠카로드는 기획~MVP 완성까지 실제로는 **약 5일**(개발 회고 1장 참고)이 걸렸다. 코딩 자체는 병목이 아니었다는 뜻이므로, 이 문서는 맠카로드에서 이미 검증된 설계(2장 "그대로 승계하는 결정")는 재론의 없이 확정하고, **이번 프로젝트에서만 새로 필요한 결정**(3장)에 집중한다. 아직 정할 수 없는 것은 결론을 억지로 내리지 않고 **"확인 필요"로 명시**해, 착수 전에 반드시 채워야 할 목록(6장)으로 모아둔다.

---

## 1. 한 줄 정의

> 유튜버 '똥그리오'가 일본 곳곳을 다니며 소개한 "또 먹고 싶은" 맛집과, 서울의 디저트 맛집을 모아 지도 위에서 영상과 함께 보는 팬메이드 지도 앱.

핵심 사용 시나리오 (맠카로드의 killer point를 지역만 바꿔 계승):
- "이번에 일본 이 동네 가는데, 똥그리오가 다녀간 맛집이 있나?" → 앱을 켠다 → 지도에서 해당 지역 마커를 본다 → 영상을 본다 → 갈지 말지 결정한다.
- "서울에서 디저트 먹고 싶은데 똥그리오가 추천한 곳이 어디였지?" → 지역 필터를 '서울'로 바꾼다 → 목록에서 고른다.

> **주의**: "똥그리오"는 이 문서·프로젝트 내부에서만 쓰는 작업용 이름이다. 맠카로드의 결론 J(앱 이름·광고는 채널 소유자 허락 이후에만 켠다)를 그대로 승계하므로, 실제 앱 이름/스토어 등록명/광고 활성화는 **소유자 허락 전까지는 코드명**으로 진행한다.

## 3. 이번 프로젝트에서 새로 결정하는 것

### 결론 A. 지도 SDK: 네이버맵 → **Google Maps로 교체 (확정)**

네이버맵/카카오맵은 한국 중심 서비스라 일본 지역 상세 지도·랜드마크 데이터 품질이 떨어진다. `google_maps_flutter` 공식 패키지를 쓴다.

**주의**: Google Maps Platform은 사용량 기반 과금이다(월 무료 크레딧 제공, 초과 시 과금). 네이버맵을 무료로 쓰던 것과 다르므로 다음을 반드시 세팅한다:
- Google Cloud Console에서 API 키 애플리케이션 제한(Android 패키지명 + SHA-1) 설정
- 예산 알림(Billing Alert) 설정
- 개발 중에는 요청 수를 늘리는 실기기 반복 테스트를 과도하게 하지 않는다

### 결론 B. 지오코딩: 네이버 지오코딩 → **Google Geocoding API로 교체 (확정)**

`upload_to_firestore.py`의 구조(엑셀 → API로 조회수/좌표 자동 채움 → Firestore)는 그대로 두고 `geocode()` 함수만 교체한다. 일본 주소는 건물명 기반인 경우가 많아 좌표 오차 가능성이 있음 — 기존 스크립트가 "엑셀에 이미 lat/lng가 있으면 존중"하는 구조라 수동 보정 여지가 이미 확보되어 있다.

### 결론 C. 길찾기 딥링크: **구글맵 기본, 서울은 확인 필요**

일본에서는 네이버지도/카카오맵이 사실상 쓰이지 않으므로 구글맵 딥링크 하나로 충분하다. 다만 서울 디저트 항목은 한국 유저 대상이라 네이버지도가 여전히 익숙할 수 있다 — **6장 열린 질문 참고**, 결정 전까지는 "구글맵 하나로 통일"을 잠정안으로 둔다 (지역별로 버튼을 다르게 보여주는 것보다 구현이 단순하고, MVP 원칙 "일단 지도에 잘 모아놓는 것"에 부합).

### 결론 D. 데이터 모델: category는 유지, **region 필드 신설 (확정)**

일본 맛집과 서울 디저트는 "장소 유형"(category) 축과는 별개로 "지역"이라는 축이 하나 더 필요하다. category 하나만으로는 지역 구분이 안 되므로 필드를 분리한다.

```
region: japan / seoul   (신규 필드 — 지도 초기 카메라 위치, 지역 필터칩에 사용)
category: restaurant / cafeDessert / attraction / bakery   (기존 그대로 유지)
```

지도 화면에 **지역 필터칩(일본/서울)**을 추가하고, 선택된 지역에 따라 초기 카메라 위치를 전환한다. 이 설계면 맠카로드의 카테고리 체크리스트 시트·필터칩 Row 코드를 거의 그대로 복사해 region 축 하나만 얹으면 된다.

### 결론 E. 전화 걸기: 국제 전화번호 포맷 대응 (확정)

일본 전화번호는 `+81-...` 형식이 필요하다. `ExternalLinksService.callPhone`의 `Uri(scheme: 'tel', path: phone)`은 국제 포맷 문자열을 그대로 넣어도 동작하므로 **코드 변경은 불필요**하지만, 엑셀 데이터 입력 단계에서 전화번호를 `+81` 국제 포맷으로 통일해서 입력하는 규칙을 정한다.

### 결론 F. 목표 출시: 착수일로부터 **3개월 내외 (확정, 사용자 지정)**

맠카로드가 기획~MVP까지 실측 5일이 걸렸다는 근거([개발 회고 1장](./개발회고_맠카로드.md#1-실제로-얼마나-걸렸나))가 있으므로, 3개월은 코딩 관점에서는 넉넉한 버퍼다. 실제 병목은 데이터 수집(6장 참고)과 지역 범위 미확정 상태이므로, 일정은 "코딩 기간"보다 "데이터 조사 기간"에 여유를 더 둔다 (5장 참고).

## 4. MVP 기능 명세

맠카로드는 화면1(지도)이 개발 도중 필터/정렬 기능이 계속 늘어났다([개발 회고 2장](./개발회고_맠카로드.md#2-기획서-대비-실제로-달라진추가된-것)). 이번엔 그 최종 형태를 처음부터 목표로 잡는다 — 처음부터 다 만들라는 뜻이 아니라, "화면 3개"가 아니라 "화면1(지도) + 화면2(상세) + 목록/즐겨찾기 탭 시트"의 2트랙 구조로 설계를 시작한다는 뜻이다.

### 화면 1: 메인 지도 (Home)
- GoogleMap 전체 화면 배경
- 마커: category(모양·아이콘) × region(초기 카메라 전환) 구분
- Floating 검색창 (가게 이름 검색)
- 필터: 카테고리 체크리스트 + **지역(일본/서울) 필터칩** + 조회수 임계값 칩 + 즐겨찾기 토글칩
- '목록' 버튼 → 조회수순/최신순/즐겨찾기 탭 바텀시트
- 내 위치 FAB
- 하단 고정 배너 영역 (비활성 상태로 준비)

### 화면 2: 맛집 상세 (Bottom Sheet)
- 가게명 + 즐겨찾기 별
- 조회수 (기준일 표기)
- 주소(탭=복사) / 전화(탭=걸기, `+81`/`+82` 국제 포맷 그대로)
- 영업시간 / 추천메뉴
- 길찾기 버튼: 구글맵 딥링크 (서울 네이버지도 병행 여부는 확인 필요)
- 공유 버튼
- 임베디드 유튜브 플레이어 + '유튜브에서 전체 보기' 버튼

### 명시적으로 만들지 않는 것
로그인 / 유저 평점·리뷰 / percent 순위 뱃지 / 종료 전면광고 / 다국어 UI(일본어 등, v2 이후) / 실시간 환율 표시(v2 이후) / 링크 붙여넣기 검색(v1.1 이후)

## 5. 기술 스택 & Firestore 스키마

| 영역 | 선택 | 비고 |
|---|---|---|
| 프레임워크 | Flutter (Android 우선) | 맠카로드와 동일 |
| 지도 | **google_maps_flutter** | 결론 A |
| 영상 | youtube_player_iframe | 재사용 (임베드 실패 대응 로직 그대로 이식) |
| DB | Cloud Firestore (읽기 전용) | 재사용, `region` 필드 추가 |
| 로컬 저장 | shared_preferences | 재사용 |
| 광고 | google_mobile_ads (배너, 비활성) | 재사용 |
| 지오코딩 | **Google Geocoding API** | 결론 B |
| 데이터 입력 | 엑셀 템플릿 → Python 스크립트 → Firestore | 구조 재사용, geocode() 함수만 교체 |
| 테마 색상 | 확인 필요 | 6장 참고. 맠카로드처럼 채널 고유색이 있다면 그대로 시드 컬러로 사용 |

### Firestore 스키마 (restaurants 컬렉션, 안)
```
name          가게명
address       주소
lat, lng      좌표
phone         전화번호 (국제 포맷 +81/+82)
hours         영업시간
menu          추천메뉴
videoId       유튜브 영상 ID (문서 ID로도 사용)
publishedAt   영상 등록일
viewCount     조회수
viewCountAt   조회수 기준일
category      restaurant / cafeDessert / attraction / bakery
region        japan / seoul   ← 신규
```

## 6. 착수 전 확인이 필요한 열린 질문 (미확정 — 결론 내지 않음)

이 목록이 다 채워지기 전까지는 화면1 뼈대 개발(데이터 의존 부분)에 들어가지 않는 것을 권장한다. 코드 구조는 어차피 맠카로드에서 재사용하므로 먼저 시작해도 되지만, 데이터 모델과 지도 초기 위치는 아래가 정해져야 확정된다.

1. **일본 지역 범위**: 도쿄·오사카 등 특정 도시 위주인지, 전국에 분산되어 있는지. → 지도 초기 카메라 위치, 데이터 수집 우선순위에 직결.
2. **서울 항목에도 네이버지도 길찾기를 남길지**: 결론 C의 잠정안(구글맵 통일)을 확정할지, 지역별로 버튼을 다르게 보여줄지.
3. **데이터 규모**: 맠카로드처럼 Top 100 하나로 할지, 일본/서울을 합쳐 Top 100인지 지역별로 나눌지 (데이터 소스가 두 축이라 총량이 늘어날 가능성).
4. **실제 착수일**: 5장 일정은 상대 주차로 표기했으므로, 착수일이 정해지면 역산해 날짜를 채운다.
5. **채널 허락 시도 시점**: 맠카로드처럼 "MVP를 완성한 뒤 제안"할지, 데이터 수집 단계에서 미리 컨택할지.
6. **테마 색상**: 채널 고유 색상이 있는지, 없다면 맠카로드(#00887A)와는 다른 색으로 구별할지.

## 7. 대략적 일정 (착수일 기준 상대 주차, 3개월 = 약 12주)

| 주차 | 내용 |
|---|---|
| 1주차 | 레포 세팅, CLAUDE.md 작성, Google Maps API 키 발급+결제계정+제한 설정, **6장 열린 질문 확정** |
| 2주차 | 화면1 뼈대 포팅 (맠카로드 코드 기반, 지도만 Google Maps로 교체) |
| 3주차 | 화면2 상세 시트 포팅 (거의 그대로 재사용) |
| 4주차 | 필터(카테고리+지역+조회수)/정렬/즐겨찾기 포팅 |
| 5~9주차 | **데이터 수집** (일본+서울 리스트 조사, 엑셀 입력, 지오코딩 스크립트 검증) ← 최대 리스크 구간, 버퍼 가장 두껍게 |
| 10주차 | 실기기 테스트, 아이콘/스플래시, 지인 베타 |
| 11주차 | 채널 허락 제안 메일 (시연 영상 포함) |
| 12주차 | 버퍼 + 출시 준비 |

맠카로드의 실측 개발 속도(코드 자체는 약 5일)를 감안하면 2~4주차는 상당히 여유 있는 배분이다. 남는 시간은 데이터 수집(5~9주차)으로 자연스럽게 흡수되도록 설계했다.

## 8. 리스크와 대응

| 리스크 | 대응 |
|---|---|
| Google Maps API 과금 | 키 제한(패키지명+SHA-1) + 청구 알림 설정을 1주차에 선행 |
| 일본 주소 지오코딩 오차 | 스크립트가 이미 "엑셀에 lat/lng 있으면 존중" 구조라 수동 보정 가능 |
| 데이터 소스 2배(일본+서울) | 지역별 Top 수를 낮춰 총량 조절 (6장 질문 3) |
| 지역 범위 미확정 상태로 개발 시작 | 1주차에 반드시 확정, 확정 전 데이터 의존 화면은 하드코딩 3개 마커로만 진행 |
| 채널 무응답 | 맠카로드와 동일 플랜B: 특정 채널 연상 요소를 뺀 중립 이름으로 전환 (단, 이 경우도 출시 전 재논의) |

## 9. 성공 기준 (MVP)

맠카로드와 동일한 기준을 승계한다:
1. 앱을 켜면 3초 안에 지도와 마커가 뜬다
2. 마커 탭 → 바텀시트 → 영상 재생까지 끊김 없이 된다
3. 전화(국제 포맷 포함)/길찾기/공유가 실제 기기에서 동작한다
4. 지역 필터로 일본 ↔ 서울을 전환했을 때 지도 카메라와 마커가 올바르게 바뀐다 ← 이번 프로젝트에서 새로 추가된 기준
5. 내가 직접 이 앱으로 일본 여행 또는 서울 디저트 방문지를 한 곳 고른다
