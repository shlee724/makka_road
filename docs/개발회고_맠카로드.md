## 2. 기획서 대비 실제로 달라진/추가된 것

기획서 3장은 화면1을 "검색창 + 필터칩 2개(조회수순/최신순)"로 계획했지만, 실제로는 화면1에 다음이 추가로 구현됐다 (모두 "하나씩 작게 추가"하며 자연스럽게 늘어난 것 — CLAUDE.md의 "한 번에 하나의 기능만" 규칙이 잘 지켜진 결과):

- 카테고리 체크리스트 필터 바텀시트 (`widgets/category_filter_sheet.dart`)
- 조회수 임계값 필터칩 (100만/50만/10만 이상, `widgets/map_filter_chips.dart`)
- 즐겨찾기 온오프 토글칩 (지도 위에서 즐겨찾기만 보기)
- '목록' 버튼 → 조회수순/최신순/즐겨찾기 3탭 바텀시트 (`screens/restaurant_list_sheet.dart`)
- 카테고리에 `bakery`(베이커리) 추가 (기획서는 restaurant/cafeDessert/attraction 3종만 계획)

화면3(즐겨찾기 목록)은 계획대로 별도 화면으로 만들었다가, 화면1의 '목록' 시트가 생기면서 **즐겨찾기 탭으로 통합**했다(`refactor: 즐겨찾기 화면을 목록 시트의 탭으로 통합`). 별도 화면을 먼저 만들고 나중에 통합한 순서는 나쁘지 않았다 — 두 기능이 필요한지 아닌지는 실제로 만들어봐야 명확해지는 경우가 많다.

## 3. 최종 아키텍처

```
lib/
  main.dart                      # MaterialApp, ColorScheme.fromSeed(#00887A)
  config/naver_map_key.dart      # API 키 (gitignore)
  firebase_options.dart          # flutterfire configure 생성 파일
  models/restaurant.dart         # Restaurant, RestaurantCategory(색상·아이콘 포함)
  services/
    favorites_service.dart       # shared_preferences 즐겨찾기 CRUD
    external_links_service.dart  # 전화/지도딥링크/공유/클립보드
  utils/view_count_format.dart   # 조회수 서식·색상
  widgets/
    home_top_bar.dart            # 검색창+필터버튼+필터칩 묶음 (조립만 함)
    restaurant_search_bar.dart
    filter_button.dart
    map_filter_chips.dart
    category_filter_sheet.dart
    category_marker_icon.dart    # 마커 아이콘 위젯(NOverlayImage로 변환됨)
    pill_button.dart
  screens/
    home_screen.dart             # 지도 화면 — 상태 관리 총괄 (StatefulWidget)
    restaurant_detail_sheet.dart # 상세 바텀시트
    restaurant_list_sheet.dart   # 목록/즐겨찾기 탭 바텀시트
```

파일당 300줄 규칙(CLAUDE.md)은 끝까지 지켜졌다 — 가장 큰 파일이 `restaurant_detail_sheet.dart`(293줄)다. **위젯을 기능 단위로 쪼개 놓으니 300줄을 넘기 전에 자연스럽게 분리 지점이 보였다.** 이 폴더 구조(`screens/widgets/services/models/utils/config`)는 그대로 다음 프로젝트에 복사해서 시작해도 된다.

### 핵심 설계 패턴 (재사용 가치 높음)

1. **상태는 부모(HomeScreen)가 갖고, 자식 위젯은 콜백만 받는다.** `_selectedCategories`, `_minViewCount`, `_favoritesOnly` 등 모든 필터 상태는 `home_screen.dart`의 `_HomeScreenState`에 있고, `HomeTopBar`/`MapFilterChips`/`FilterButton`은 값과 `onChanged` 콜백만 받는 순수 표시 위젯이다. Provider 없이 setState만으로 화면1의 복잡한 필터 조합(카테고리 × 조회수 × 즐겨찾기)이 문제없이 돌아갔다 — CLAUDE.md의 "setState 우선" 원칙이 화면1 규모에서는 충분했다는 실증.

2. **마커 아이콘은 (카테고리, 즐겨찾기여부) 조합별로 한 번만 만들어 캐싱한다.** `_markerIconKey(category, isFavorite)`로 키를 만들어 `Map<String, NOverlayImage>`에 저장 (`home_screen.dart:69-82`). 지도가 준비될 때 8개(카테고리 4종 × 즐겨찾기 2종) 이미지를 한 번만 생성하고, 이후 마커 추가/즐겨찾기 토글 시 기존 이미지를 재사용한다. 매번 위젯을 새로 렌더링하지 않아 성능이 안정적이다.

3. **가시성은 마커를 지웠다 다시 그리지 않고 `isVisible`만 바꾼다.** `_applyFilters()`가 필터 결과에 따라 기존 마커의 `setIsVisible()`만 토글한다. 필터를 자주 바꾸는 UX(칩 여러 개)에서 마커를 매번 삭제/재생성하면 깜빡임과 성능 저하가 생기는데, 이 패턴으로 회피했다.

4. **외부 연동은 "앱 우선, 실패 시 웹 폴백" 패턴을 서비스 레이어에 통일.** `ExternalLinksService._launchAppOrWeb()`이 `canLaunchUrl(앱딥링크)`를 먼저 확인하고 안 되면 웹 URL로 연다. 네이버지도/카카오맵 둘 다 이 함수 하나를 재사용한다.

## 4. 트러블슈팅 (다음 프로젝트에서 미리 피할 수 있는 것들)

- **유튜브 임베드가 일부 영상에서 무한 로딩된다.** 채널이 임베딩을 막아둔 영상은 공식 에러 콜백 없이 "Please try again later" 화면에서 멈춘다. → `_playbackTimeout`(8초) 안에 재생 가능 상태(`cued` 이상)에 도달하지 못하면 실패로 간주하고 안내 문구로 대체 (`restaurant_detail_sheet.dart:40-74`). 2025년 유튜브 정책 변경 이후 `origin`을 `youtube-nocookie.com`으로 지정하는 것도 재생 성공률을 높였다.
- **유튜브 웹뷰가 세로 드래그 제스처를 가로챈다.** `enableFullScreenOnVerticalDrag: false` + `VerticalDragGestureRecognizer`를 시트와 경쟁시켜서, 영상 위에서도 바텀시트를 끌어내릴 수 있게 했다 (`restaurant_detail_sheet.dart:200-209`).
- **하단 배너가 시스템 내비게이션 바에 가려진다.** `bottomNavigationBar`를 회색 `Container`로 감싸고, 그 안에 `SafeArea`로 실제 배너 영역을 넣어 색은 시스템 바 뒤까지 이어지되 콘텐츠는 안전 영역 안에 배치했다 (`home_screen.dart:262-267`).
- **키보드가 올라올 때 지도가 울렁거린다.** `Scaffold(resizeToAvoidBottomInset: false)`로 해결. 검색창이 화면 상단에 있어 키보드에 가려질 걱정이 없어서 가능했던 선택.
- **Firestore 필수값 누락 시 앱 전체가 죽는다.** `Restaurant.fromFirestore`가 일부 필드(`phone`, `hours`, `menu`)는 빈 문자열을 허용하도록 방어했지만, `name`/`videoId`처럼 없으면 의미가 없는 필드는 그대로 캐스팅 예외를 낸다 → 업로드 스크립트(`upload_to_firestore.py`) 쪽에서 필수값 검증을 먼저 하는 게 근본 대책이었다.

## 5. 데이터 파이프라인 (그대로 재사용 가능)

```
엑셀 템플릿(사람이 노란 칸만 입력: name/address/videoId/category/phone/hours/menu)
  → upload_to_firestore.py
      1) 카테고리 유효성 검사 (Dart enum과 이름이 정확히 일치해야 함)
      2) YouTube Data API v3로 videoId → viewCount/publishedAt 자동 채움 (50개씩 배치, 무료 쿼터로 충분)
      3) 네이버 지오코딩 API로 address → lat/lng 자동 채움 (엑셀에 이미 좌표가 있으면 존중)
  → Firestore에 batch 업로드 (문서ID = videoId → 재실행해도 중복 없이 덮어씀, 월 1회 조회수 갱신도 동일 스크립트 재실행으로 해결)
```

이 구조 자체(엑셀 → 자동 보강 스크립트 → Firestore, 문서ID로 videoId 사용해 재실행 안전)는 매우 잘 작동했다. 다음 프로젝트에서는 **지오코딩 함수만 Google Geocoding API로 교체**하면 나머지 구조를 그대로 쓸 수 있다.

**보안 관행도 그대로 유지할 것**: `upload_to_firestore.py`, `serviceAccountKey.json`, `lib/config/naver_map_key.dart`는 모두 `.gitignore`에 등록되어 있고 실제로 한 번도 커밋된 적이 없다(`git log --all -- upload_to_firestore.py` 결과 없음으로 확인됨). API 키가 스크립트 안에 하드코딩돼 있긴 하지만 로컬 전용 파일이라 노출 위험은 없다 — 이 패턴(파이프라인 스크립트는 gitignore, 앱 코드의 키만 별도 config 파일로 분리)을 다음 프로젝트에도 그대로 적용한다.
