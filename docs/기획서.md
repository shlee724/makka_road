## 0. 이 문서의 성격

아이디어 스케치(raw)에서 이미 내려진 결론은 그대로 존중하고, **열려 있던 고민에는 결론을 확정**하여 더 이상 고민하지 않도록 만든 문서. 개발 중 방향이 흔들릴 때 이 문서로 돌아온다.

스케치에서 이미 확정된 것 (변경 없음):
- MVP의 본질은 **"일단 지도에 잘 모아놓는 것"**
- killer point 1 ("우리 동네에도 맠카님이 다녀간 곳이 있나?")을 메인으로 설계
- MVP 데이터는 **인기 영상 Top 100을 수동 입력**
- 영상은 임베디드로

## 1. 한 줄 정의

> 대전 맛집 유튜버 '맠카'가 다녀간 맛집만 모아, 지도 위에서 영상과 함께 보는 팬메이드 지도 앱.

핵심 사용 시나리오 (killer point 1):
"내가 사는 동네 근처에도 맠카님이 가보신 곳이 있나?" → 앱을 켠다 → 지도에서 내 주변 마커를 본다 → 마커를 눌러 영상을 본다 → 갈지 말지 결정한다.

## 2. 열려 있던 고민들에 대한 확정 결론

### 결론 A. 임베디드 vs 외부 링크 → **하나의 UI로 둘 다 해결 (확정)**

스케치의 "2개 버전을 다 만들어봐야 할 것 같다"는 계획은 **폐기**한다. 바텀시트 안에 임베디드 플레이어(youtube_player 계열 패키지)를 넣고, 플레이어 바로 아래에 `유튜브 앱에서 전체 보기` 버튼 하나를 추가하면 두 방식의 장점이 모두 확보된다. A/B 버전을 따로 만드는 것은 1인 개발에서 개발 기간을 두 배로 만드는 가장 흔한 함정이다.
- 참고: 스케치의 "AI로 영상에서 사진 추출 → 지도앱처럼 꾸미기" 아이디어는 좋은 v2 방향이지만 MVP에서는 완전히 잊는다.

### 결론 B. 조회수 percent 순위 → **MVP에서 제외 (확정)**

분포표 기반 percent 계산은 오버엔지니어링이다. MVP에서는:
- 상세 뷰에 **조회수 원본 숫자** 표시 (예: "조회수 51,322회 · 2026.6 기준")
- 목록/필터에서 **조회수순 정렬** 제공
  이 두 가지로 "유명한 맛집을 나도 가본다"는 재미의 대부분이 구현된다. percent 뱃지("상위 5% 인기 맛집" 등)는 v2 아이디어 목록으로 이동.

### 결론 C. 유저 평점 시스템 → **제외 (확정)**

스케치에서도 "조금 고려해보기"였던 항목. 평점을 넣는 순간 로그인, 유저 DB, 어뷰징 관리, 악성 리뷰 신고 처리가 전부 딸려온다. 이 앱의 정체성은 "맠카님의 큐레이션"이지 리뷰 플랫폼이 아니다. 유저의 선호 표현은 **즐겨찾기(별)** 하나로 충분하다.

### 결론 D. 즐겨찾기 저장 위치 → **서버가 아닌 폰 로컬 (확정)**

Firestore 스키마에 있던 "즐겨찾기 여부" 필드는 삭제한다. 즐겨찾기는 shared_preferences(또는 Hive)로 **기기 로컬에 저장**한다. 이 결정 하나로 로그인/회원가입 기능 전체가 불필요해지고, MVP 난이도가 크게 내려간다.

### 결론 E. 네이버/카카오 평점 가져오기 → **하지 않는다 (확정)**

스케치 메모("안되겠지?")의 답: 안 된다가 맞다. 크롤링은 양 사 약관 위반 소지가 있고, 공식 API로도 평점은 제공되지 않는다. 대신 상세 뷰의 `길찾기` 버튼이 네이버지도/카카오맵 딥링크로 열리므로, 유저는 한 탭이면 그쪽 평점을 볼 수 있다. 이 정도면 UX상 충분하다.

### 결론 F. 검색창 형태 → **Floating Search Bar (확정)**

AppBar 없이, Stack + Positioned로 지도 위에 떠 있는 검색창. 네이버지도/카카오맵과 동일한 문법이라 유저에게 가장 익숙하다.

### 결론 G. 바텀시트 동작 → **DraggableScrollableSheet, "내리면 접히고, 더 내리면 닫힘" (확정)**

스케치의 "닫는 느낌? 내리는 느낌?" 고민의 답: Flutter 기본 위젯인 DraggableScrollableSheet가 이 동작(중간 높이 ↔ 전체 ↔ 닫기)을 기본 제공한다. 커스텀으로 만들지 말 것.

### 결론 H. 조회수 업데이트 → **MVP는 월 1회 수동, v1.x에서 YouTube Data API 자동화 (확정)**

스케치의 "1달마다 주기적인 수동 업데이트"는 MVP 단계에서 옳은 선택이다. 다만 알아둘 것: YouTube **공식** Data API의 videos.list는 영상 50개씩 조회수를 한 번에 반환하므로, Top 100 전체가 API 호출 2번이면 끝난다(무료 쿼터로 매우 충분, 약관 문제 없음). 출시 후 Python 스크립트 하나로 자동화하는 것을 v1.x 과제로 잡는다. "최신 영상은 조회수 미표시" 규칙은 그대로 유지.

### 결론 I. 종료 전면광고 → **MVP에서 제외, 배너만 (확정)**

먹방로드를 따라 종료 전면광고를 넣는 계획은 보류한다. 이유: ① AdMob 정책상 앱 종료 시점 전면광고는 심사에서 문제가 되는 사례가 잦다 ② 첫인상이 중요한 팬메이드 앱에서 UX를 깎는다. 하단 고정 배너 하나로 시작하고, 전면광고는 수익 데이터를 본 뒤 판단.

### 결론 J. 광고 자체의 시점 → **맠카님 허락 이후에만 켠다 (확정)**

타인의 채널명과 콘텐츠로 만든 앱에 허락 전부터 광고를 붙이면 제안의 진정성이 훼손되고 법적으로도 위험하다. 개발 기간 동안 앱 이름은 **코드명(예: "로드맵D")**으로 진행하고, AdMob 연동 코드는 만들어두되 꺼둔다. 맠카님이 허락하면 이름과 광고를 켠다. 임베디드 재생 자체는 유튜브 공식 임베드 기능이라 문제없다.

### 결론 K. killer point 2 (링크 붙여넣기 → 가게 표시) → **v1.1 최우선 (확정)**

스케치에서는 "사람들 반응 보고 결정"이었지만, 구현 난이도를 보면 결정이 쉽다. DB에 이미 영상 링크(videoId)가 있으므로, **클립보드의 유튜브 URL에서 videoId를 파싱해 Firestore에서 일치하는 가게를 찾는 것**이 전부다. 예상 작업량 1~2일. "사진 자동 추출로 지도 꾸미기"(수 주 소요)보다 압도적으로 가성비가 좋으므로, 업데이트 순서는 **링크 붙여넣기 먼저**로 확정한다. 이것이 스케치에서 말한 "매직"이고, 맠카님께 제안할 때도 가장 임팩트 있는 데모가 된다.

## 3. MVP 기능 명세 (이것만 만든다)

### 화면 1: 메인 지도 (Home)
- NaverMap 전체 화면 배경 (flutter_naver_map)
- 맛집 마커 표시 (NMarker) — Firestore에서 로드, category별 색상·아이콘 구분
- Floating 검색창 (가게 이름 검색)
- 필터 칩 2개: `조회수순` / `최신순` (누르면 목록 시트로 정렬 결과 표시)
- 우측 하단 내 위치 FAB (locationTrackingMode 사용, 위치 권한 요청 포함)
- 하단 고정 배너 영역 (허락 전까지는 빈 공간 또는 앱 로고)

### 화면 2: 맛집 상세 (Bottom Sheet)
- DraggableScrollableSheet
- 가게명 + 즐겨찾기 별(로컬 저장)
- 조회수 (기준일 표기, 최신 영상은 미표기)
- 주소 (탭 → 복사) / 전화번호 (탭 → 전화 걸기)
- 영업시간 / 추천메뉴
- `길찾기` 버튼 → 네이버지도·카카오맵 딥링크
- `공유` 버튼
- 임베디드 유튜브 플레이어 + `유튜브에서 전체 보기` 버튼

### 화면 3: 즐겨찾기 목록
- 로컬에 저장된 별표 가게 리스트 → 탭하면 지도에서 해당 마커로 이동
- (스케치에는 없었지만 즐겨찾기가 있는 이상 목록이 없으면 반쪽 기능이라 추가. 단순 ListView라 저비용)

### 명시적으로 만들지 않는 것 (v2 이후 대기 목록)
유저 평점 / percent 순위 뱃지 / AI 사진 추출 지도 꾸미기 / 종료 전면광고 / 로그인 / 조회수 자동 갱신(→v1.x) / 링크 붙여넣기 검색(→v1.1)

## 4. 기술 스택 확정

| 영역 | 선택 | 비고 |
|---|---|---|
| 프레임워크 | Flutter (Android 우선) | iOS는 출시 후 검토. 개발자 계정 비용·심사 이중부담 회피 |
| 지도 | flutter_naver_map | 마커·내위치 트래킹 내장. 네이버클라우드 콘솔에서 키 발급 필요 |
| 영상 | youtube_player_iframe (또는 youtube_player_flutter) | 유튜브 공식 임베드 기반 |
| DB | Cloud Firestore (읽기 전용 공개 데이터) | 앱 배포 없이 가게 추가/수정 가능. 보안규칙: read만 허용 |
| 로컬 저장 | shared_preferences | 즐겨찾기 |
| 광고 | google_mobile_ads (배너, 비활성 상태로 준비) | 허락 후 활성화 |
| 데이터 입력 | Google Sheets에 정리 → Python 스크립트로 Firestore 업로드 | 100개 수기 입력을 앱 밖에서 편하게 |
| 테마 색상 | `#00887A` (맠카님 테마 색상) | 앱 전체 시드 컬러로 사용 |

### Firestore 스키마 (restaurants 컬렉션)
```
name          가게명
address       주소
lat, lng      좌표 (마커용 — 스케치에 없었지만 필수 필드)
phone         전화번호
hours         영업시간
menu          추천메뉴
videoId       유튜브 영상 ID (링크 대신 ID로 저장 — v1.1 매칭에 유리)
publishedAt   영상 등록일 (조회수 표시 여부 판단에 사용)
viewCount     조회수
viewCountAt   조회수 기준일
category      장소 유형: restaurant(식당) / cafeDessert(카페·디저트) / attraction(관광명소)
```
즐겨찾기 필드는 삭제(로컬 저장으로 이동). "영상 링크로 업데이트 기간 분류" 메모는 publishedAt 필드로 해결.

**category 추가 배경**: 맠카님 영상에는 식사 맛집 외에 카페·디저트, 가끔 관광명소도 섞여 있다. "맛집만 모은 지도"라는 1장의 정의는 유지하되, 방문지 성격이 다르면 마커 색상·아이콘으로 구분해 지도를 봤을 때 바로 감이 오게 한다 (식당 = 테마색 `#00887A`, 카페·디저트 = 앰버, 관광명소 = 보라).

## 5. 일정 (2026.7 ~ 12)

전제: **7/2 ~ 7/16은 완전 휴식.** 이 기간의 계획은 "계획하지 않는 것"이다. 굳이 뭔가 하고 싶어지면 맠카 영상을 그냥 재미로 보는 것까지만 — 그게 곧 데이터 수집이기도 하다. 아래 일정은 전체 기간의 절반만 실작업으로 채운 여유 있는 일정이므로, 중간에 컨디션이 안 좋은 주가 생겨도 12월 출시에는 지장이 없게 설계했다.

### Phase 0 — 휴식 (7/2 ~ 7/16)
아무것도 안 함. 노트도 안 봄.

### Phase 1 — 워밍업 학습 (7/17 ~ 8/10, 하루 1~2시간이면 충분)
스케치의 "해야할 것 순서"를 그대로 따르되, 각 위젯을 **본 프로젝트와 무관한 미니 앱으로 하나씩** 만들어본다:
1. 노마드코더 복습 (1주)
2. flutter_naver_map 공식 샘플 실행 — 지도 띄우기 + 마커 1개 + 내 위치 (3~4일)
3. youtube_player로 영상 하나 재생하는 화면 (1~2일)
4. DraggableScrollableSheet 데모 (1~2일)
5. Firestore에서 데이터 3개 읽어와 리스트로 뿌리기 (2~3일)
6. 병행: GitHub에서 Flutter 지도앱 프로젝트 1~2개 코드 구경 (스케치의 "남의 코드 보기" 실천)

이 6개가 다 돌아가면 맠카로드는 사실상 "조립"만 남는다.

### Phase 2 — MVP 개발 (8/11 ~ 9/30, 약 7주)
- 8월 3주: 프로젝트 세팅, 네이버맵 키, 화면 1 뼈대 (지도 + 마커 하드코딩 3개)
- 8월 4주 ~ 9월 2주: 화면 2 (바텀시트 + 임베디드 영상 + 전화/길찾기/공유 연동)
- 9월 3주: Firestore 연동, 검색/정렬, 즐겨찾기
- 9월 4주: 화면 3, 버퍼 주간
- 개발 방식: 위젯 트리(스케치의 Stack 구조)를 기준으로 Claude Code와 페어코딩. 막히면 "이 화면의 위젯 트리는 이런데 여기서 X가 안 된다"처럼 구조를 들고 질문하면 가장 빠르다.

### Phase 3 — 데이터 & 폴리싱 (10월)
- 맠카 채널 인기 영상 Top 100 시트 정리 (조회수는 유튜브에 그대로 보이므로 수집 자체는 단순 노동. 하루 20개씩 5일)
- Python 업로드 스크립트로 Firestore 투입
- 실기기 테스트, 아이콘/스플래시, 지인 3~5명 베타

### Phase 4 — 제안 & 심사 (11월)
- **MVP가 실제로 돌아가는 시연 영상**을 찍어 비즈니스 메일 발송 (스케치의 제안서 초안 활용, 발송 전 이 대화에서 검수 1회)
- 메일의 설득 포인트: "만들겠다"가 아니라 "이미 만들었다 + 이름·광고는 허락 전까지 안 붙였다"
- 답장 대기 중 Play Console 등록, 비공개 테스트 트랙 심사 진행 (허락 여부와 무관하게 진행 가능 — 이름만 코드명 유지)
- 허락 시: 앱명 '맠카로드' 변경, AdMob 활성화, 수익 배분 협의

## 7. 성공 기준 (MVP)
